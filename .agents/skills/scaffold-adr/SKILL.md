# scaffold-adr — Architecture Decision Records (Create, Supersede, Audit)

Manage the `.agents/architecture/decisions/` folder: create new ADRs with required sections and auto-numbering, supersede existing ADRs without violating immutability, and audit the whole folder for missing sections / dangling links / stale proposals / revisit triggers.

> **BEFORE EXECUTING:** Read [`SKILL-implementation.md`](SKILL-implementation.md) (sibling file) for the full step-by-step logic — required sections per mode, numbering rules, supersede mechanics (the *only* allowed mutation of an existing ADR), audit classification rules, and error handling. This page is orientation only.

---

## When to Use

| Situation | Mode |
|---|---|
| You made a non-obvious technical choice and need to record it | **New** |
| You're replacing a past decision; the prior ADR should be marked superseded | **Supersede** |
| You want to know if any ADRs are malformed, stale, or have broken links | **Audit** |
| You're doing an architecture review or onboarding handoff | **Audit** |
| You set a "Revisit when" trigger months ago and wonder if it's time | **Audit** |

---

## Layer-Presence Guard

The skill refuses to run if `.agents/architecture/decisions/` does not exist. The architecture layer (which contains ADRs) is opt-in at init — if it's not present, the user should re-run the init skill and select the layer first.

## ADR Format — YAML Frontmatter

Every ADR generated by this skill uses YAML frontmatter (consistent with `scaffold-context` and `scaffold-architecture`):

```yaml
---
adr-number: 0001
title: Use pgvector for embedding storage
status: Proposed   # Proposed | Accepted | Rejected | Superseded | Deprecated
date: 2026-05-13
supersedes: null   # or "ADR-0003" if this supersedes a prior decision
superseded-by: null  # set on the prior ADR when a successor is accepted
revisit-when: "if vector dimension exceeds 4096 or we adopt a managed vector DB"
generated-by: scaffold-adr
verified-against: <git HEAD SHA at creation time>
verified-at: 2026-05-13T00:00:00Z
---
```

After the frontmatter, the ADR body uses standard sections: Title (H1) / Owner + Deciders / Context / Decision / Alternatives considered / Consequences (Positive / Negative / Follow-up) / Revisit when. The body does not repeat metadata from the frontmatter.

---

## Immutability Contract

**Accepted ADRs are immutable.** Once an ADR has `status: Accepted` in its frontmatter, the skill will never edit it — with one explicit exception: **Supersede mode** updates two YAML frontmatter fields on the prior ADR:

- `status: Accepted` → `status: Superseded`
- `superseded-by: null` → `superseded-by: "ADR-<NNNN>"`

Nothing else in the prior ADR changes. The body is never modified.

A `status: Proposed` ADR is still mutable by hand-edit, but this skill does not edit them either — proposals evolve via comments, discussion, and re-drafts that overwrite the proposal in place by the human, not the agent.

## Mode 1: New

Guided creation of a fresh ADR with all required sections.

- Walks the user through: **Title** → **Context** → **Decision** → **Alternatives considered** → **Consequences (positive/negative/follow-up)** → **Revisit when**.
- Auto-numbers: scans `decisions/` for highest existing `NNNN-` prefix; uses `max + 1`, zero-padded to 4 digits. Never reuses a number, even if a prior ADR was deleted (numbers are append-only).
- Generates filename: `NNNN-<kebab-from-title>.md`. Trims to 60 chars max; rejects empty / non-ASCII-after-slug-strip titles.
- Pre-fills `status: Proposed` and `date` as today in the YAML frontmatter.
- Captures git HEAD SHA for `verified-against`; `revisit-when` frontmatter field captures the trigger the user provides.
- **Refuses to overwrite** an existing file at the target path. If a collision happens (race condition with a manual edit), reports the conflict and asks the user to choose a new title.
- Optionally appends an entry to `llms.txt` under `adr-dir:` if the user opts in.

**When to pick it:** Any time you make a choice with multi-quarter consequences. The bar is low — better an over-documented decision than a re-litigated one.

## Mode 2: Supersede

Create a new ADR that supersedes an existing one. **The only allowed mutation of a prior ADR.**

**Atomicity guarantee:** Supersede mode checks preconditions before writing anything. Either both the new ADR is written and the target's frontmatter is updated, or neither happens. The sequence is:

1. **Precondition check (dry-run):** Read the target ADR's frontmatter. Confirm it parses as YAML, that `status` is exactly `Accepted`, that `superseded-by` is null (prevent double-supersede), and that the replacement text is uniquely matchable. If any check fails, abort with a clear error — no files are written.
2. **Write the new ADR** (runs the Mode 1 guided flow, with the new ADR's `supersedes` frontmatter field set to `"ADR-<target-NNNN>"`).
3. **Edit the target's frontmatter** — change `status: Accepted` → `status: Superseded` and `superseded-by: null` → `superseded-by: "ADR-<new-NNNN>"`. Nothing else in the target changes.

If the superseded ADR's `status` is not `Accepted`, refuses with an explanation and asks the user to clarify intent.

**When to pick it:** A past decision no longer holds — vendor change, scale threshold crossed, regulatory requirement shifted. Always supersede; never edit-in-place.

## Mode 3: Audit

Read-only report on the state of `.agents/architecture/decisions/`. **Never writes.**

**Sub-agent fan-out:** Audit mode dispatches one Haiku sub-agent per ADR (cap 20, oldest first) in a single parallel batch. Each sub-agent validates that ADR's YAML frontmatter and body sections. The main Sonnet agent collects results and then performs cross-ADR checks that require the full inventory.

Per-ADR checks (each Haiku sub-agent):
- **Required frontmatter fields present**: `adr-number`, `title`, `status`, `date`. Flag missing fields.
- **Status value valid**: one of `Proposed`, `Accepted`, `Rejected`, `Superseded`, `Deprecated`. Flag invalid.
- **Date present and ISO-formatted**.
- **Required body sections present**: `## Context`, `## Decision`, `## Alternatives considered`, `## Consequences`, `## Revisit when`.
- **`revisit-when` non-empty for Accepted ADRs older than 12 months**. Empty = ossification risk.
- **Stale proposals**: `status: Proposed` ADRs with no git activity (or `date` field) for >90 days.

Cross-ADR checks (main Sonnet agent):
- **Duplicate `adr-number` values**: two files with the same number.
- **Numbering gaps**: missing `adr-number` between two consecutive values (informational; not always an error).
- **Dangling `superseded-by` links**: `superseded-by: "ADR-NNNN"` pointing at a number that doesn't exist.
- **Orphan supersede claims**: `supersedes: ADR-X` where ADR-X does not have `superseded-by` pointing back.

Produces a structured report; ends with `AskUserQuestion`: "Run scaffold-adr in Supersede or New mode for any flagged ADRs?"

**When to pick it:** Quarterly, during architecture reviews, or before handing off ownership of a service.

## What Gets Created / Updated

### New mode
- New file at `.agents/architecture/decisions/NNNN-<kebab>.md` (with YAML frontmatter)
- (Optional) `llms.txt` entry under `adr-dir:` — only if user opts in

### Supersede mode
- New file at `.agents/architecture/decisions/NNNN-<kebab>.md` (the new ADR, with YAML frontmatter)
- Two-field YAML frontmatter edit to the superseded ADR (`status` and `superseded-by` only; body unchanged)

### Audit mode
- **Nothing written.** Output only: per-ADR report (from Haiku sub-agents) + cross-ADR findings (from main agent).

### Never touched by this skill
- Any ADR's body (paragraphs, headings, lists) — ever
- `.agents/architecture/decisions/0000-template.md` (the template itself — `tidy-scaffold` removes if unused, but this skill leaves it alone)
- `.agents/architecture/` Mermaid diagrams (those are `scaffold-architecture`'s domain)
- `.agents/architecture/README.md`
- Anything outside `.agents/architecture/decisions/`, except the optional `llms.txt` ADR entry append

## Self-Management Contract

- **Reads `project_context.md` before every run** — extracts Owner field as the default for the ADR Owner field.
- **Refuses to run if `.agents/architecture/decisions/` is absent** — directs user to re-run init.
- **Never reuses ADR numbers** — append-only, even if a prior ADR was deleted.
- **Never edits an accepted ADR's body** — only the two frontmatter fields (`status`, `superseded-by`) in Supersede mode.
- **Never auto-creates ADRs in batch** — one ADR per skill invocation. Decisions deserve deliberate framing.
- **Never invents context or alternatives** — if the user gives sparse answers, asks follow-ups; never fabricates rationale.
- **Never reads secret files** — `.env`, `*.pem`, `*.key`, etc.
